Class-> In Java, a method's signature is defined by its name and the number, type, and order of itsparameters. It does not include the return type or access modifiers.-> Java class can have multiple methods of same name also known as overloading in Java but the onlycondition is that the signature of the methods should be different.-> In a class you can also have multiple constructors with different signatures, and they can call eachother by using simple this() or this(arguments) if some parameters are there.-> There is something called type casting in class also which is just converting one class reference toother but the condition is that in case of class type casting the cast classes are within the sameinheritance hierarchy. More about it in Inheritance.Inheritance------------> Java has public, protected and private access modifier.-> public: The element is accessible from anywhere within the Java project.This includes within the same class, within the same package, and from classes in different packages.-> protected: The element is accessible within the same package and by subclasses (even if thosesubclasses are in different packages).-> private: The element is accessible only within the declared class itself. It cannot be accessedfrom outside the class, even by subclasses or classes within the same package.-> if parent class have public or protected access methods or variable than only they can be inherited-> if parent method has protected modifier then a lower or equal access modifier can be assigned tochild class only not higher access modifier.-> Overriding decorator is not necessary in java but important to signify that even the signatureof methods in parent or child class are same the method is override. This is done for increasingreadability of user and for the compiler to check the parent class and can't find a method with anidentical signature to the one marked with @Override, it throws an error.-> If signatures are different then its overloading not overriding.-> There is some thong called sealed class in Java which avoids other subclasses to inherit class andonly the permitted subclasses are allowed to inherit the class.-> Also, there's a class called final class which means that this class cannot be further inherited anyclass.-> If you want to call parent constructor then use super() or super(arguments) if there are some parametersto be passed.-> As, you know about typecasting happens in class only in same inheritance hierarchy, there are two typesof type casting:    1) Implicit Upcast (Widening Casting)        -> Here we convert a smaller subtype or lower precision type to a larger supertype or high precision        class type automatically like no explicit code needed like (class_name)object_name.    2) Explicit Downcast (Narrowing Casting)        -> Explicit down cast means converting a larger (supertype) reference or higher-precision type to a        smaller (subtype) or lower-precision type manually, using a cast operator.    3)   For Ex:            super_class obj = new sub_class(); // Upcast (subclass instance assign to superclass object)            sub_class new_obj = (sub_class) obj; // Downcast (explicitly converting the object of superclass                                                to subclass so, it can be referenced to subclass new object.)Polymorphism-------------> Seriously its just have a simple meaning many forms of same thing.-> It is simple achieved by method overloading and overriding like same method name but different work.-> Two type of Polymorphism :    1) Compile-Time Polymorphism (Method Overloading)        -> This type of polymorphism is resolved at compilation time as when we doing method overloading        with same name methods and different signatures then with this different signatures if we call this        method the compiler knows what method we are calling.        -> One more example is use of '+' in number additions and strings additions here, operator        overloading is used.    2) Runtime Polymorphism (Method Overriding)        -> Also knows as runtime polymorphism this is achieved through method overriding because at runtime        when the instance of the class is created then only java get to know which class method we are calling        the subclass or the parent class.        -> For example in case of Shape example the user is asked for a option for shape type and then based        on that option class instance is created, supposed square is choose then the method area and perimeter        of square class will be called which were override from the superclass shape.        -> This is polymorphism nothing else, same name different functions.Abstraction------------> Abstract simple here is template like the abstract class and method are generally referred as atemplate for other classes.-> If a subclass in inheriting the abstract class then every abstract method defined in the abstract classneed to be implemented in the subclass.-> If you want to only use some of the methods from the abstract superclass then your subclass will need tobe an abstract class also, but for a normal class all abstract methods needs to implemented.-> You cannot instantiate the abstract class, but you can have the object of it like:                abstract_class obj = new non_abstract_class();-> In case you want to call some method directly from the abstract class then use 'static' keyword inthe method definition as static method can be called directly using the abstract class object asinstallation is not allowed for abstract methods. Ex:                abstract_class.static_method();Interface----------> It is just the brother of abstraction but only difference is that any methods or variable define in it arepublic, static and final means they cannot be changed not even the signature of methods and value of variables.-> An Interface class can only inherit by other interface class.-> A normal class cannot inherit the interface class but can implement the interface class using'implements' keyword.-> A normal class can implements more than 1 interface class(just after implements add all interface classseparated by comma).-> While all the method defined in the interface class doesn't have body but if some method with body needsto be defined in interface class then 'default' keyword used in the starting of that method which can printand written the value both. This method can be overridden in the subclass also.-> Similar to abstract class here interface class also cannot be instantiated, but it's object can be usedfor calling static method defined in the interface class.Blocking Queue---------------> A blocking queue is java disallows value like null and throws null pointer exemption as null servesa special marker to signal the failure of certain operations like poll, when an element that is polledis not available within the specified wait time as another thread is using it.-> BQ has a parameter maximum capacity limit which signifies the max number of elements queue can hold anymoment, ny attempt to add element after capacity reach will lead to blocking. The default value ofqueue is Integer.MAX_VALUE, if not specified.-> While serving the functionality of consumer and producer, BQ also implements the Collection Interfaceof Java due to which you can also use functions like : add(e), remove(e), contains(e), size(),isEmpty(), iterator(), and clear() function where e is either element or object.-> BQ are thread safe and ensure atomicity in operation like put() or take() through internal lockingmechanisms or other concurrency controls. However, when we talk about the operations like addAll,containAll, retainAll, or removeAll, they might not be atomic. As there are chances that these 'All' typeoperations will partially succeed like addAll elements will be able to add the elements but after someelements an exemption might be thrown.-> When a producer thread calls the put(E e) method to add an element to the queue and the queue isalready at its full capacity, the put() method will block the calling thread. This means theproducer thread will pause its execution and wait. When a consumer thread successfully removes anelement, the BlockingQueue internally uses a mechanism (often involving Condition objects associatedwith a Lock) to signal any waiting producer threads. This signal unblocks one of the waiting producerthreads, allowing it to resume execution and successfully add its element to the now-available spacein the queue.-> Two major examples of BQ are: ArrayBlockingQueue and LinkedBlockingQueue.-> Usage and limitations:    - The internal locking mechanisms that ensure thread safety can become a headache in highly concurrent    applications where many threads want the queue access.    - when many thread are trying to access the queue at the same time the throughput of the application might get    affected as queue synchronization mechanism resist all the thread to access the queue at the same time.    - Implementing a shutdown mechanism using special end-of-stream or poison objects like null requires an additional    logic to handle these objects properly. This can complicate the producer-consumer logic as these objects will    require additional conditional statement.    - Not suitable for all concurrent applications as some application might require ordered non-blocking or some    special access patterns non-blocking behaviour like in banking applications there might be an access pattern that    needs to be keep in mind while making blocking queue for threads in it.    - Operations involving multiple elements, such as addAll and removeAll, are not guaranteed to be performed    atomically, which might lead to partial updates and require additional handling to ensure consistency.
